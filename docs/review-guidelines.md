# マージレビュー観点

レビューでは、コードの好みよりも、仕様、動作、運用、セキュリティへの影響を優先して確認する。
指摘は、再現できる事実、Specとの差分、将来の運用リスクに基づいて書く。

## レビューの入口

レビュー担当者は、最初にPR本文を見る。
関連Issue、関連Spec、Milestone、確認結果、未確認事項が書かれていないPRは、実装を見る前に差し戻す。

次に、関連Specを読む。
PRの差分がSpecに書かれた範囲を超えている場合、実装を広げる前にSpec更新またはIssue分割を求める。

## 共通観点

| 観点 | 確認すること |
|---|---|
| Spec整合 | 受け入れ条件を満たしているか |
| 変更範囲 | PRが1つのIssueに収まっているか |
| 動作確認 | テストまたはcurlなどの確認結果があるか |
| 失敗時の挙動 | validation error、例外、再起動時の扱いが説明されているか |
| ドキュメント | README、Spec、Runbook、ADRが必要に応じて更新されているか |
| セキュリティ | Secret値、過剰な権限、認証情報が含まれていないか |
| 運用 | ログ、Probe、設定値、障害時の確認方法が考慮されているか |

## APIレビュー

API変更では、リクエスト、レスポンス、HTTP status、validation errorを確認する。
MVPではOpenAPIを手書きしない場合でも、SpecにAPIの期待値を残す。

`POST /logs` では、ログスキーマの必須項目がSpecと一致していることを確認する。
`GET /logs` では、メモリ保存であることが利用者に誤解されないようにする。

## データスキーマレビュー

ログスキーマを変えるPRでは、`SPEC.md` と該当する `specs/` を更新する。
将来のBigQueryやPub/Subを見越す場合でも、Phase 1の実装に不要な項目を必須にしない。

`agent_id`、`actor_user_id`、`tenant_id` は混同しない。
Agentそのもの、人間ユーザー、組織境界は別の概念として扱う。

## Kubernetesレビュー

Kubernetes manifestでは、Deployment、Service、ConfigMap、Secret、Probeの責務を分けて確認する。
Secret値を直接commitしていないことを必ず確認する。

Probeのpathは、アプリの実装と一致している必要がある。
Serviceのport、targetPort、containerPortも同時に確認する。

## GCPレビュー

GCP関連の変更では、作成するリソース名、リージョン、IAM権限、削除方法を確認する。
手作業で作るリソースは、`deploy/gcloud-commands.md` または関連ドキュメントに記録する。

IAMは、作業を通すために広い権限を付けない。
必要な権限がまだ分からない場合は、IssueまたはADRに未決事項として残す。

## セキュリティレビュー

Secret、API Key、個人情報、認証トークンはcommitしない。
例示が必要な場合は、`secret.example.yaml` やダミー値を使う。

外部公開に関わるPRでは、認証、IP制限、Cloud Armor、ログ保持、監査ログへの影響を確認する。
Phase 1で対応しない場合でも、対象外としてSpecに残す。

## ドキュメントレビュー

ドキュメント変更では、実装済みのことと予定を区別する。
まだ存在しないコマンドやファイルを、すでに使えるものとして書かない。

手順は、開発者が上から順に実行できる粒度にする。
前提条件がある場合は、コマンドの前に書く。

## 指摘の優先度

| Priority | 意味 |
|---|---|
| P0 | mergeしてはいけない重大な問題 |
| P1 | merge前に直すべき問題 |
| P2 | できれば同じPRで直す問題 |
| P3 | 後続Issueで扱ってよい改善 |

レビューコメントでは、優先度と理由を書く。
単なる好みの指摘は、Blockingにしない。

## レビュー完了条件

レビューは、次の状態になったら完了とする。

- Blockingな指摘が解消している。
- PR本文の確認項目が埋まっている。
- 必要なテストまたは動作確認が通っている。
- 未確認事項が残る場合、後続IssueまたはSpecに移されている。
